TemplateMonster code style guide

This document contains code style methodologies that should be used by TemplateMonster team members. It contains our best practices and recommendations from respectful professional teams all over the world.

Please note. Style guide is under construction.

We code for our customers. Make sure your code is easy to read, edit and support. Consider this your main goal.

General rules and recommendations

Omit resource protocol

Integrating resources from the external servers, omit protocols from the URL(http: or https:)

									<script src="//www.google.com/js/gweb/analytics/autotrack.js"></script>
								
									.example {
										background: url(//www.google.com/images/example);
									}

Indentation

Indent only using tabs. Do not use spaces. Tabs provide more flexibility while using different code editors.

Formatting

Each element should be placed on a single line (exception for the inline elements). Always use indentation to show correct elements nesting

Markup rules

Creating webpage markup, along with the main code style guide principles you should remeber that parts of your code can be used in other projects by yourself or your team members. Make sure block elements are as independent as possible. Follow these rules:

  • Each page element markup and style rules should be independent and can be used anywhere on the page without losing styles.
  • Assign classes to key elements to avoid selector nesting
  • You can use short names for the elements and classes. Make sure your abbreviations are readable and easy to understand

For example a newsletter block: 

									

										

											
Newsletter

										

										

											

												

													Please input your email address

CSS rules for newsletter block:

									.module__newletter{}
									.module__newletter .mod_header{}
									.module__newletter .mod_header h3{}

									.module__newletter .mod_content{}

									.form__newsletter{}
									.form__newsletter p{}
									.form__newsletter p i{}
									.form__newsletter .input__email{}
									.form__newsletter .btn{}
									.form__newsletter .btn:hover{}

Abbreviations

You can use shortands and abbreviations to reduce class names and selectors. make sure your abbreviations are readable and easy to understand. For example

  • mod- module
  • btn- button
  • add- additional
  • bg- background
  • col- column
  • img- image
  • vid- video
  • scr- screen

HTML

HTML validation

HTML markup should be valid. Use this tool to validate. There should be no errors in our markup.
Exceptions are validation errors in CMS's core files.

Capitalization

Do not use capitalization it element names, class names, IDs, color HEX codes etc. All elements should be lowercase. Exception - CDATA

									
								
									color: #e5e5e5;

Doctype

Use HTML5 syntax to define doctype

									<!DOCTYPE html>

Semantics

Make sure to use semantically correct HTML elements.

Heading h1 - h5
Paragraph p
Makred italic em (i -for visual marking, not logical)
Marked bold strong (b -for visual marking, not logical)
Quotes cite - small blocks, inline elements.
blockquote - quote blocks.
Lists ol / ul
Definition lists dl > dt + dd

Type Attributes

Do not use type attributes for style sheets (unless not using CSS) and scripts (unless not using JavaScript).

Multimedia Fallback (Alt Attributes)

Provide alternative contents for multimedia. Define alt attribute values for any multimedia element you use

ID and class usage

Use ID's only to define key elements of the page or elements used as initialization selectors in JS scripts.

Commenting

HTML markup blocks should be separated from each other and provided with comments where necessary.

CSS

CSS validation

CSS styles should be valid. Use this tool to validate.

Formatting

Remember, your code should be easy to manage.

  • Each selector should be on a single line
  • Each property should be placed on a single line (exception: single rule for a single selector)
  • Style declaration should be indented to one tab from the selector
  • In case of using multiple selectors, each selector should be on a single line
  • Do not use multiple IDs in style declarations
									.class,
									tag,
									#id{
										color: #fff;
										font-size: 12px;
										width: 100%;
									}

Selector naming rules

Selector names should be logical and clear. Selector names should be as short as possible but as long as necessary. Use general names. Consider your code to be used in other projects.

									/* Not recommended: meaningless */
									#yee-1901 {}

									/* Not recommended: presentational */
									.button-green {}
									.clear {}
								
									/* Recommended: specific */
									#gallery {}
									#login {}
									.video {}

									/* Recommended: generic */
									.aux {}
									.alt {}

Keywords

To add more logic to the selector names use keywords. Keywords should be related to the functionality of the selector. For example: module, widget, cart, checkout, page

									.block{}

									.widget{}

									.cart{}

									.button{}

Modifiers

Modifiers used to assign styles to some condition of the selected element:

									.block .block__hover{}

									.widget .widget__active{}

									.cart .cart__empty{}

									.button .button__hover{}

Modifiers also can be used to assign styles to the child elements:

Separators

In selector names please use the following separators:

  • "__" - for modifiers
  • "_" - for child elements
  • "-" - to separate words in class names

Assigning classes you need to avoid adding dozens of different classes to a single element. Reduce number of classes if possible without affecting code flexibility and ease.

Type selectors

Do not use tag selectors with classes or ids. 

									/* Not recommended: meaningless */
									ul#example {}
									div.error {}
								
									/* Recommended: specific */
									#example {}
									.error {}

Nesting

Assigning CSS rules use nested selectors carefully. Creating long nesting chains can make your code hard to read. Avoid using more than 3 elements in nested selectors. Add classes to the key markup elements to reduce nisted selectors. 

									.module__modifier{}
									.module__modifier .mod_header{}
									.module__modifier .mod_header h3{}

									.form__modifier .input__email{}
									.form__modifier .btn{}
									.form__modifier .btn:hover{}

Units

Do not add units if value is zero.

									margin: 0;
									padding: 0;

Rule separation

Separate rules by new line.

									body {
									  margin: auto;
									  width: 50%;
									}

Exclusion: if there is a single rule for a selector

									html { background: #fff; }

Commenting

Use comments to separate style declaration sections. To add more logic to the comments use three variants:

1. Generic style declaration sections

								/*========================================================
								                    1. Typography
								=========================================================*/

								/*========================================================
								                    2. Main layout styles
								=========================================================*/

2. More specific sections

								/* Product listings 
								========================================================*/

								/* Checkout Page 
								========================================================*/

3. Specific elements, inline comments

								//Product image

								//Post title

4. Comments for logically independent modules/elements

								/* QR Code */

								/* Back to top */

								/* Modal */

Highly recommended to add table of contents to the top the LESS/CSS file. Only generic section names should be added to the TOC. For example: 

								/* Content:
								  1. Global
								  2. Basic elements/styles
								  3. Typography
								  4. Lists and tags
								  5. Main layout elements
								  6. Main layout styles
								  7. Error styles
								  8. Media query styles
								  9. Utility classes
								  10. Site Maintenance masking
								*/

PHP

Clean Code

Themes must not have any PHP notices, warnings, or errors - please develop with errors enabled. In php.ini make sure to have display_errors=On and error_reporting=E_ALL & ~E_STRICT

Single and Double Quotes

Use single and double quotes when appropriate. If you're not evaluating anything in the string, use single quotes. You should almost never have to escape quotes in a string, because you can just alternate your quoting style, like so:

									echo '<a href="/static/link" title="Yeah yeah!">Link name</a>';
									echo "<a href='$link' title='$linktitle'>$linkname</a>";

When a literal string itself contains apostrophes, it is permitted to demarcate the string with quotation marks or "double quotes". This is especially useful for SQL statements:

									$sql = "SELECT `id`, `name` from `people` WHERE `name`='Fred' OR `name`='Susan'";

Indentation

Your indentation should always reflect logical structure. Use real tabs and not spaces, as this allows the most flexibility across editors.

									[tab]$foo   = 'somevalue';
									[tab]$foo2  = 'somevalue2';
									[tab]$foo34 = 'somevalue3';
									[tab]$foo5  = 'somevalue4';

For associative arrays, values should start on a new line. Note the comma after the last array item: this is recommended because it makes it easier to change the order of the array, and makes for cleaner diffs when new items are added.

									$my_array = array(
									[tab]'foo'   => 'somevalue',
									[tab]'foo2'  => 'somevalue2',
									[tab]'foo3'  => 'somevalue3',
									[tab]'foo34' => 'somevalue3',
									);

Rule of thumb: Tabs should be used at the beginning of the line for indentation, while spaces can be used mid-line for alignment.

Variable Substitution

Variable substitution is permitted using either of these forms:

									$greeting = "Hello $name, welcome back!";

									$greeting = "Hello {$name}, welcome back!";

Brace Style

Braces shall be used for all blocks in the style shown here:

									if ( condition ) {
										func1();
										func2();
									} elseif ( condition2 && condition3 ) {
										func3();
										func4();
									} else {
										default_func();
									}

Braces should always be used, even when they are not required:

									if ( condition ) {
										func0();
									}

									if ( condition ) {
										func1();
									} elseif ( condition2 ) {
										func2a();
										func2b();
									}

									foreach ( $items as $item ) {
										process_item( $item );
									}

No Shorthand PHP Tags

Important: Never use shorthand PHP start tags. Always use full PHP tags

Correct:

									<?php ... ?>
									<?php echo $var; ?>

Incorrect:

									<? ... ?>
									<?= $var ?>

Remove Trailing Spaces

Remove trailing whitespace at the end of each line of code. Especially after the closing PHP tag at the end of a file.

Space Usage

Always put spaces after commas, and on both sides of logical, comparison, string and assignment operators.

									x == 23
									foo && bar
									! foo
									array( 1, 2, 3 )
									$baz . '-5'
									$term .= 'X'

Put spaces on both sides of the opening and closing parenthesis of if, elseif, foreach, for, and switch blocks.

									foreach ( $foo as $bar ) { ...

When defining a function, do it like so:

									function my_function( $param1 = 'foo', $param2 = 'bar' ) { ...

When calling a function, do it like so:

									my_function( $param1, func_param( $param2 ) );

When performing logical comparisons, do it like so:

									if ( !$foo ) { ...

When type casting, do it like so:

									foreach ( (array) $foo as $bar ) { ...

									$foo = (boolean) $bar;

When referring to array items, only include a space around the index if it is a variable, for example:

									$x = $foo['bar']; // correct
									$x = $foo[ 'bar' ]; // incorrect

									$x = $foo[0]; // correct
									$x = $foo[ 0 ]; // incorrect

									$x = $foo[ $bar ]; // correct
									$x = $foo[$bar]; // incorrect

Naming Conventions

Use lowercase letters in variable and function names (never camelCase). Separate words via underscores. Don't abbreviate variable names un-necessarily; let the code be unambiguous and self-documenting.

									function some_name( $some_variable ) { [...] }

Class names should use capitalized words separated by underscores. Any acronyms should be all upper case.

									class Simple_Class { [...] }
									class Extend_Class extends Simple_Class { [...] }

Files should be named descriptively using lowercase letters. Hyphens should separate words.

									my-plugin-name.php

Class file names should be based on the class name with class- prepended and the underscores in the class name replaced with hyphens, for example Basic_Category becomes:

									class-basic-category.php

This file-naming standard is for all current and new files with classes.

Self-Explanatory Flag Values for Function Arguments

Prefer string values to just true and false when calling functions.

									// Incorrect
									function eat( $what, $slowly = true ) {
									...
									}
									eat( 'mushrooms' );
									eat( 'mushrooms', true ); // what does true mean?
									eat( 'dogfood', false ); // what does false mean? The opposite of true?

Since PHP doesn't support named arguments, the values of the flags are meaningless, and each time we come across a function call like the examples above, we have to search for the function definition. The code can be made more readable by using descriptive string values, instead of booleans.

									// Correct
									function eat( $what, $speed = 'slowly' ) {
									...
									}
									eat( 'mushrooms' );
									eat( 'mushrooms', 'slowly' );
									eat( 'dogfood', 'quickly' );

Ternary Operator

Ternary operators are fine, but always have them test if the statement is true, not false. Otherwise, it just gets confusing. (An exception would be using ! empty(), as testing for false here is generally more intuitive.)

For example:

									// (if statement is true) ? (do this) : (else, do this);
									$musictype = ( 'jazz' == $music ) ? 'cool' : 'blah';
									// (if field is not empty ) ? (do this) : (else, do this);

Yoda Conditions (Recommendation)

									if ( true == $the_force ) {
										$victorious = you_will( $be );
									}

When doing logical comparisons, always put the variable on the right side, constants or literals on the left.

In the above example, if you omit an equals sign (admit it, it happens even to the most seasoned of us), you'll get a parse error, because you can't assign to a constant like true. If the statement were the other way around ( $the_force = true ), the assignment would be perfectly valid, returning 1, causing the if statement to evaluate to true, and you could be chasing that bug for a while.

But Yoda conditions for <, >, <= or >= are significantly more difficult to read and are best avoided.

String Concatenation

Strings must be concatenated using the "." operator. A space must always be added before and after the "." operator to improve readability:

									$company = 'Template' . ' ' . 'Monster';

When concatenating strings with the "." operator, it is encouraged to break the statement into multiple lines to improve readability. In these cases, each successive line should be padded with single tab:

									$sql = "SELECT `id`, `name` FROM `people` "
										. "WHERE `name` = 'Susan' "
										. "ORDER BY `name` ASC ";

Commenting

To ensure the possibility of phpDocumentor usage for building documentation, commenting in PHP should be done using DocComment/DocBlock syntax. This is a special type of comment which starts with /**, ends with */ and may contain any number of lines in between. Every line should start with an asterisk, which is aligned with the first asterisk of the opening clause.

								/** Single line comment */

								/**
								 * Multiline comment
								 */

DocBlock is a DocComment containing a single PHPDoc and represents the basic in-source representation. PHPDoc is a section of documentation which provides information on several aspects of Structural Elements.

Tags must be present in the DocBlocks. Tags are a type of specialized information (meta-data) about the associated element. @return is a tag in the example below.

								/**	
								  * Returns the name of this object.
								  *
								  * @return string
								  */
  • phpDocumentor glossary
  • List of tags

Example of function DocBlock: 

								 /**
								  * A summary informing the user what the associated element does.
								  *
								  * A *description*, that can span multiple lines, to go _in-depth_ into the details of this
								  * element and to provide some background information or textual references.
								  *
								  * @param string $myArgument With a *description* of this argument, these may also
								  *    span multiple lines.
								  *
								  * @return void
								  */
								  function myFunction($myArgument){}

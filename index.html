TemplateMonster code style guide

This document contains code style methodologies that should be used by TemplateMonster team members. It contains best practices and respectful team's recommendations.
Please note. Style guide is under construction.

We code for our customers. Make sure your code easy to read, edit and support. Consider this your main goal.

General rules and recommendations

Omit resource protocol

Integrating resources from the external servers, omit protocols from the URL(http: or https:)

									<script src="//www.google.com/js/gweb/analytics/autotrack.js"></script>
								
									.example {
										background: url(//www.google.com/images/example);
									}

Indentation

Indent only using tabs. Do not use spaces.

Formatting

Each element should be placed on a single line (exception for the inline elements). Always use indentation to show correct elements nesting

HTML

HTML validation

HTML markup should be valid. Use this tool to validate. There should be no errors in our markup.
Exceptions are validation errors in CMS's core files.

Capitalization

Do not use capitalization it element names, class names, IDs, color HEX codes etc. All elements should be lowercase. Exception - CDATA

									
								
									color: #e5e5e5;

Doctype

Use HTML5 syntax to define doctype

									<!DOCTYPE html>

Semantics

Make sure to use semantically correct HTML elements.

Heading h1 - h5
Paragraph p
Makred italic em (i -for visual marking, not logical)
Marked bold strong (b -for visual marking, not logical)
Quotes cite - small blocks, inline elements.
blockquote - quote blocks.
Lists ol / ul
Definition lists dl > td + dd

Type Attributes

Do not use type attributes for style sheets (unless not using CSS) and scripts (unless not using JavaScript).

Multimedia Fallback (Alt Attributes)

Provide alternative contents for multimedia. Define alt attribute values for any multimedia element you use

ID and class usage

Use ID's only to define key elements of the page or elements used as initialization selectors in JS scripts.

CSS

CSS validation

CSS styles should be valid. Use this tool to validate.

Formatting

Remember, your code should be easy to manage.

  • Each selector should be on a single line
  • Each property should be placed on a single line
  • Style declaration should be indented to one tab from the selector
  • In case of using multiple selectors, each selector should be on a single line
  • Do not use multiple IDs in style declarations
									.class,
									tag,
									#id{
										color: #fff;
										font-size: 12px;
										width: 100%;
									}

Selector names

Selector names should be logical and clear. Selector names should be as short as possible but as long as necessary. Use general names. Consider your code to be used in other projects.

									/* Not recommended: meaningless */
									#yee-1901 {}

									/* Not recommended: presentational */
									.button-green {}
									.clear {}
								
									/* Recommended: specific */
									#gallery {}
									#login {}
									.video {}

									/* Recommended: generic */
									.aux {}
									.alt {}

Type selectors

Do not use tag selectors with classes or ids. 

									/* Not recommended: meaningless */
									ul#example {}
									div.error {}
								
									/* Recommended: specific */
									#example {}
									.error {}

Units

Do not add units if value is zero.

									margin: 0;
									padding: 0;

Rule separation

Separate rules by new line.

									html {
									  background: #fff;
									}

									body {
									  margin: auto;
									  width: 50%;
									}
